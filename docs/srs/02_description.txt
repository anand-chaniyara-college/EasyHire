2.1 PRODUCT PERSPECTIVE
--------------------------------------------------------------------------------

2.1.1 System Context:

EasyHire is a standalone web application designed to be deployed on-premises or in 
cloud infrastructure. It interfaces with external email services but does not 
integrate with other enterprise systems in the current scope.

2.1.2 System Interfaces:

    a) Web Browser Interface:
       - Modern browsers (Chrome 90+, Firefox 88+, Edge 90+, Safari 14+)
       - HTML5, CSS3, JavaScript ES6 support required
       - Minimum resolution: 1024x768 (responsive design)

    b) Database Interface:
       - MySQL 8.0+ database server
       - JDBC connection via HikariCP connection pool
       - JPA/Hibernate ORM layer

    c) Email Interface:
       - SMTP server for outgoing emails (port 587 with TLS)
       - IMAP server for reading inbox (port 993 with SSL)
       - Default configuration: Gmail SMTP/IMAP

    d) WebSocket Interface:
       - STOMP over WebSocket for real-time notifications
       - SockJS fallback for browser compatibility

2.1.3 Hardware Interfaces:
    - No direct hardware interfaces
    - Standard server hardware for deployment
    - Client: Any device with modern web browser

    - HTTP/HTTPS for web traffic (port 80/443)
    - WebSocket for real-time notifications
    - SMTP (port 587) for sending emails

2.1.6 Memory Constraints:

    - File upload limit: 16MB per file

2.3.2 User Class Details:

    a) Default-Admin:
       - Frequency: 1 per system
       - Technical Level: High
       - Permissions: Full system access
       - Responsibilities:
         * Manage other admins
         * System configuration
         * All Admin capabilities

    b) Admin:
       - Frequency: 1-5 per organization
       - Technical Level: Medium-High
       - Permissions: All except admin management
       - Responsibilities:
         * User management
         * Department management
         * Interview status decisions
         * All Senior HR capabilities

    c) Senior-HR:
       - Frequency: 2-10 per organization
       - Technical Level: Medium
       - Permissions: Candidate status, emails, employment
       - Responsibilities:
         * Change candidate status
         * Send interview invitations
         * Employ candidates
         * Recall candidates
         * All Junior HR capabilities

    d) Junior-HR:
       - Frequency: 5-20 per organization
       - Technical Level: Low-Medium
       - Permissions: Read/write vacancies, interviews
       - Responsibilities:
         * Create/edit positions
         * Create interviews
         * Download CVs
         * Generate reports
         * Filter and view data

    e) Department-Head:
       - Frequency: 1 per department
       - Technical Level: Low-Medium
       - Permissions: Department-specific data
       - Responsibilities:
         * View department candidates
         * Change interview statuses
         * View department dashboard

    f) Interviewer:
       - Frequency: Many per department
       - Technical Level: Low
       - Permissions: Assigned interviews only
       - Responsibilities:
         * View assigned candidates
         * Submit reviews
         * View interview details

    g) Candidate (Public):
       - Frequency: Unlimited
       - Technical Level: Low
       - Permissions: Public pages only
       - Responsibilities:
         * View job listings
         * Submit applications

2.5 DESIGN AND IMPLEMENTATION CONSTRAINTS
--------------------------------------------------------------------------------

2.5.1 Technical Constraints:

    C1: Must use Spring Boot 2.7.x framework
    C2: Must use MySQL as primary database
    C3: Must use Thymeleaf for server-side rendering
    C4: Must use Spring Security for authentication
    C5: File uploads limited to 16MB per file
    C6: Only PDF and Word documents for CV
    C7: Sequential interview stages only
    C8: Single-tenant deployment model
    C9: Server-side session management

2.5.2 Regulatory Constraints:

    C10: Password must be encrypted (BCrypt)
    C11: Audit trail for status changes
    C12: Role-based access control required

2.5.3 Development Constraints:

    C13: Maven-based build system
    C14: Must include unit tests for services
    C15: Must follow MVC architecture pattern

2.6 ASSUMPTIONS AND DEPENDENCIES
--------------------------------------------------------------------------------

2.6.1 Assumptions:

    A1: Users have stable internet connectivity
    A2: Users have modern web browsers with JavaScript enabled
    A3: Email server (SMTP/IMAP) is available and configured
    A4: MySQL database server is provisioned before deployment
    A5: System clock is synchronized for date/time operations
    A6: Maximum concurrent users: 100
    A7: Data retention: All data retained indefinitely
    A8: Single geographic location deployment
    A9: English language interface only

2.6.2 Dependencies:
